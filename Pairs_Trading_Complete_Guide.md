# หลักการเทรด Pairs Trading ฉบับสมบูรณ์

## ทุกอย่างที่ต้องรู้ก่อนเริ่มเทรด — จากทฤษฎีถึงปฏิบัติจริง

---

## บทที่ 1: Pairs Trading คืออะไร

### 1.1 แนวคิดพื้นฐาน

Pairs Trading คือกลยุทธ์ที่ทำกำไรจากความสัมพันธ์ระหว่างสินทรัพย์ 2 ตัว ไม่ใช่จากทิศทางตลาด

สมมติเหรียญ A กับ B เคลื่อนไหวคล้ายกันมาตลอด 6 เดือน วันหนึ่ง A พุ่งขึ้นผิดปกติแต่ B ไม่ขึ้นตาม เกิด "ช่องว่าง" ระหว่างสองตัว เราเชื่อว่าช่องว่างนี้จะปิดลง จึงเข้าเทรด: Short ตัวที่แพงเกิน (A) และ Long ตัวที่ถูกเกิน (B) พร้อมกัน

เมื่อทั้งสองกลับมาเคลื่อนไหวคล้ายกันอีกครั้ง ช่องว่างปิด เราปิดทั้ง 2 ขาพร้อมกัน กำไรมาจากส่วนต่างที่ช่องว่างหดตัว

### 1.2 ทำไมถึงได้ผล

เหรียญที่อยู่ในกลุ่มเดียวกัน (เช่น Layer-1 ทั้งคู่ หรือ Meme Coin ทั้งคู่) มักถูกขับเคลื่อนด้วยปัจจัยเดียวกัน เช่น Sentiment ตลาด, ข่าว Sector, การเคลื่อนไหวของ BTC เมื่อเกิดการถ่างออกชั่วคราว มักเป็นเพราะปัจจัยเฉพาะตัวที่ไม่ยั่งยืน ราคาจึงมีแนวโน้มกลับมาหากัน

### 1.3 Market Neutral คืออะไร

เนื่องจากเรา Long 1 ตัวและ Short 1 ตัวพร้อมกัน ถ้าตลาดร่วงทั้งหมด ขา Short กำไร ขา Long ขาดทุน แต่ถ้าความสัมพันธ์ยังอยู่ ผลลัพธ์สุทธิจะขึ้นกับ "ช่องว่างหดลงหรือไม่" ไม่ใช่ "ตลาดขึ้นหรือลง" ทำให้กลยุทธ์นี้ทำกำไรได้ทั้งตลาดขาขึ้นและขาลง

### 1.4 ตัวอย่างง่ายๆ

```
วันที่ 1: ETH = $3,400  SOL = $140
วันที่ 5: ETH = $3,600  SOL = $140  ← ETH พุ่ง SOL ไม่ตาม (ถ่างออก)

เราเข้าเทรด:
  Short ETH $500 ที่ $3,600
  Long SOL  $500 ที่ $140

วันที่ 12: ETH = $3,450  SOL = $148  ← กลับมาบรรจบ

ปิดทั้ง 2 ขา:
  Short ETH กำไร: (3,600 - 3,450) / 3,600 × $500 = +$20.83
  Long SOL กำไร:  (148 - 140) / 140 × $500       = +$28.57
  รวม: +$49.40 (ก่อนหัก Fee + Funding)
```

## บทที่ 2: อินดิเคเตอร์ทั้ง 6 ตัว — อธิบายละเอียด

### 2.1 Correlation (ค่าสหสัมพันธ์ Pearson)

**คืออะไร:** วัดว่าเหรียญ 2 ตัวเคลื่อนไหวไปในทิศทางเดียวกันมากแค่ไหน

**ค่าที่เป็นไปได้:**
- +1.0 = เคลื่อนไหวเหมือนกันทุกประการ (A ขึ้น B ขึ้นเสมอ)
- 0.0 = ไม่มีความสัมพันธ์
- -1.0 = เคลื่อนไหวตรงข้ามกันทุกประการ

**วิธีคำนวณ:**

```
Correlation = Σ[(Aᵢ - μA)(Bᵢ - μB)] / (n × σA × σB)

โดย:
  Aᵢ, Bᵢ = ราคาปิดรายวันของเหรียญ A และ B
  μA, μB = ค่าเฉลี่ยราคา
  σA, σB = ส่วนเบี่ยงเบนมาตรฐาน
  n = จำนวนวัน
```

**เกณฑ์:**

| Correlation | ความหมาย | เหมาะกับ Pairs Trading? |
|---|---|---|
| ≥ 0.85 | สูงมาก | ดีมาก — คู่หลัก |
| 0.80–0.84 | สูง | ดี — ใช้ได้ |
| 0.70–0.79 | ปานกลาง | ระวัง — ต้องมี Cointegration ด้วย |
| < 0.70 | ต่ำ | ไม่แนะนำ |

**ข้อควรระวัง:**
- ใช้ Rolling Correlation (เช่น 90 วัน) ไม่ใช่ค่าเดียวตลอด
- ถ้า Correlation ลดลงต่ำกว่า 0.50 ระหว่างถือ → ออกทันที
- Correlation สูงไม่ได้หมายความว่า "ถ่างแล้วจะกลับมา" ต้องตรวจ Cointegration ด้วย

**Period ที่ใช้:** คำนวณจากราคาปิดรายวัน ย้อนหลัง 90 วัน ตรวจซ้ำทุก 2 สัปดาห์

---

### 2.2 Cointegration / Hurst Exponent

**ปัญหาของ Correlation:** Correlation บอกแค่ว่า "เคลื่อนไหวคล้ายกัน" แต่ไม่บอกว่า "ถ้าถ่างออกจะกลับมาหากัน" เหรียญ 2 ตัวอาจมี Correlation สูงแต่พอถ่างออกแล้วไม่กลับมาอีกเลย

**Cointegration บอกอะไร:** บอกว่า Spread ระหว่าง 2 เหรียญมีคุณสมบัติ Mean-Reverting หรือไม่ คือ "เมื่อถ่างออกไป จะมีแรงดึงให้กลับมาหาค่ากลาง" ซึ่งเป็นหัวใจของ Pairs Trading

**วิธีทดสอบ (Hurst Exponent):**

Hurst Exponent (H) คือตัวเลขที่บอกพฤติกรรมของ Spread:

```
H < 0.5 → Mean-Reverting (ถ่างแล้วกลับ) → ดีสำหรับ Pairs Trading ✓
H = 0.5 → Random Walk (เดินสุ่ม) → ไม่เหมาะ ✗
H > 0.5 → Trending (ถ่างแล้วถ่างต่อ) → อันตราย ✗✗
```

**วิธีคำนวณ Hurst:**

```
1. คำนวณ Spread series: S = Price_A - (β × Price_B)
2. หาค่าเฉลี่ย Spread
3. คำนวณ Cumulative Deviation จากค่าเฉลี่ย
4. R = Max(Cumulative) - Min(Cumulative)   ← Range
5. S = Standard Deviation ของ Spread       ← Scale
6. H = ln(R/S) / ln(N)                     ← Hurst
```

**เกณฑ์:**

| Hurst | ความหมาย | Action |
|---|---|---|
| 0.20–0.40 | Mean-Reverting แรง | ดีมาก — คู่เกรด A |
| 0.40–0.49 | Mean-Reverting เบา | ดี — ใช้ได้ |
| 0.50 | Random Walk | ข้ามไป |
| > 0.50 | Trending | ห้ามเทรด! |

---

### 2.3 Spread (ส่วนต่างราคา)

**คืออะไร:** คือ "ช่องว่าง" ระหว่างเหรียญ 2 ตัว หลังปรับด้วย Hedge Ratio

**สูตร:**

```
Spread = Price_A − (β × Price_B)

โดย β = Hedge Ratio (อธิบายในข้อ 2.6)
```

**ทำไมต้องคูณ β:**

เพราะเหรียญ 2 ตัวมีราคาและความผันผวนต่างกัน เช่น ETH = $3,400 กับ SOL = $140 ถ้าเอาราคามาลบกันตรงๆ ไม่มีความหมาย β ช่วยปรับสัดส่วนให้เปรียบเทียบได้อย่างยุติธรรม

**วิธีใช้:** นำ Spread มาพล็อตเป็นกราฟเส้น ถ้ามันเคลื่อนไหวขึ้นลงรอบค่ากลาง (Mean) แสดงว่าคู่นี้มีคุณสมบัติ Mean-Reverting ดี ถ้า Spread ถ่างออกมาก → โอกาสเข้าเทรด ถ้า Spread กลับมาที่ค่ากลาง → โอกาสปิดทำกำไร

---

### 2.4 Z-Score ★ (ตัวจับสัญญาณหลัก)

**คืออะไร:** วัดว่า Spread ปัจจุบันห่างจากค่าปกติกี่ส่วนเบี่ยงเบนมาตรฐาน (SD) เป็นตัวเลขมาตรฐานที่ใช้ตัดสินใจเข้า-ออก

**สูตร:**

```
Z-Score = (Spread ปัจจุบัน − Mean ของ Spread) / SD ของ Spread

โดย:
  Mean = ค่าเฉลี่ยเคลื่อนที่ 60 วันของ Spread
  SD = ส่วนเบี่ยงเบนมาตรฐาน 60 วันของ Spread
```

**วิธีอ่านค่า:**

```
Z-Score = 0    → Spread อยู่ที่ค่าปกติ (ไม่มีโอกาส)
Z-Score = +1.0 → ถ่างออก 1 SD (ยังไม่พอเข้า)
Z-Score = +2.0 → ถ่างออก 2 SD → สัญญาณ Short Spread ★
Z-Score = +3.0 → ถ่างออก 3 SD → Stop Loss!
Z-Score = -2.0 → ถ่างออก 2 SD อีกฝั่ง → สัญญาณ Long Spread ★
```

**ทำไมใช้ ±2.0 เป็นจุดเข้า:**

ในการแจกแจงปกติ ค่าที่เกิน ±2 SD มีโอกาสเกิดแค่ ~5% ของเวลาทั้งหมด หมายความว่า Spread ถ่างออกผิดปกติจริงๆ และมีโอกาสสูงที่จะกลับมา

**Period:** ใช้ 60 วันในการคำนวณ Mean และ SD อัปเดต Z-Score ทุกนาที (ใช้ราคาล่าสุด + cached Mean/SD)

---

### 2.5 Half-Life (เวลากลับตัว)

**คืออะไร:** วัดว่า Spread ใช้เวลากี่วันในการกลับมาหาค่ากลางครึ่งทาง

**ตัวอย่าง:** ถ้า Half-Life = 10 วัน หมายความว่า เมื่อ Spread ถ่างออก 2 SD หลังจาก 10 วัน Spread จะลดลงเหลือประมาณ 1 SD (ครึ่งทาง)

**สูตร (Ornstein-Uhlenbeck):**

```
1. คำนวณ ΔSpread = Spread(t) - Spread(t-1)
2. ทำ Regression: ΔSpread = λ × Spread(t-1)
3. Half-Life = -ln(2) / λ

เงื่อนไข: λ ต้อง < 0 (ถ้า λ ≥ 0 แปลว่า Spread ไม่กลับ → ข้ามคู่นี้)
```

**เกณฑ์:**

| Half-Life | ความหมาย | เหมาะไหม |
|---|---|---|
| < 5 วัน | กลับเร็วมาก | ⚠️ อาจไม่ทันเข้า |
| 5–15 วัน | เหมาะสม | ✅ ดี — ถือไม่นาน |
| 15–30 วัน | ค่อนข้างช้า | ✅ ใช้ได้ — ถือนานหน่อย |
| > 30 วัน | ช้าเกินไป | ❌ เงินถูก Lock นาน เสี่ยง |

**ใช้ทำอะไร:**
- กำหนด Time Stop: ถือไม่เกิน 2× Half-Life → ตัดทิ้ง
- ประมาณระยะเวลาถือ: ปกติถือ 0.5× ถึง 1.5× Half-Life
- คัดเลือกคู่: Half-Life 5–30 วันเท่านั้น

---

### 2.6 Hedge Ratio (β)

**คืออะไร:** สัดส่วนที่ต้องเทรดแต่ละขาเพื่อให้ Position เป็น Market Neutral จริงๆ

**สูตร:**

```
β = Covariance(A, B) / Variance(B)

หรือพูดง่ายๆ: β = ผลจาก Regression เส้นตรง A = α + β×B
```

**ตัวอย่าง:**

```
ถ้า β = 1.3 หมายความว่า:
  ทุกๆ $1 ที่ ETH เปลี่ยน SOL จะเปลี่ยน $1.3

ดังนั้นถ้า Long ETH $500:
  ต้อง Short SOL $500 × 1.3 = $650
  เพื่อให้ทั้ง 2 ขา "ถ่วงดุล" กัน
```

**ข้อควรระวัง:**
- β ต้อง > 0 (ถ้า ≤ 0 คู่นี้ไม่เหมาะ)
- β ต้องไม่เป็น Infinity หรือ NaN (เกิดจากราคา = 0 หรือ Variance = 0)
- β เปลี่ยนแปลงได้ตามเวลา → ตรวจสอบทุก 2 สัปดาห์
- ถ้า β เปลี่ยนเกิน 20% จากตอนเปิด → พิจารณาปิดสถานะ

## บทที่ 3: ขั้นตอนคัดเลือกคู่เหรียญ (Pair Selection)

### 3.1 ขั้นตอนที่ 1 — คัดกรองเบื้องต้น

```
เริ่มจาก: เหรียญทั้งหมดที่มี USDT-M Futures บน OKX

กรองรอบแรก:
  ✅ Volume 24h > $20,000,000 (มี Liquidity พอ)
  ✅ มี Futures Contract จริง
  ✅ ราคา > 0 (ป้องกัน Division by Zero)

ผลลัพธ์: ประมาณ 20–30 เหรียญ
```

### 3.2 ขั้นตอนที่ 2 — จับคู่ทั้งหมดที่เป็นไปได้

```
20 เหรียญ → 20 × 19 / 2 = 190 คู่
25 เหรียญ → 25 × 24 / 2 = 300 คู่
```

### 3.3 ขั้นตอนที่ 3 — กรองด้วย Indicator ทีละตัว

```
190 คู่ → กรอง Correlation ≥ 0.80 → เหลือ ~40 คู่
40 คู่  → กรอง Hurst < 0.50      → เหลือ ~25 คู่
25 คู่  → กรอง Half-Life 5-30d    → เหลือ ~15 คู่
15 คู่  → กรอง |Z-Score| ≥ 2.0    → เหลือ ~3-5 คู่ (Signal!)
```

### 3.4 ขั้นตอนที่ 4 — Safe Entry Zone

```
3-5 คู่ที่มี Signal → ตรวจ Safe Zone:

  ถ้า SL = 3.0 และ Safe Buffer = 0.5:
  SafeMax = 3.0 - 0.5 = 2.5

  |Z| ≤ 2.3  → ✅ SAFE — เปิดเต็ม 100%
  |Z| 2.3-2.5 → ⚠️ CAUTION — เปิด 50%
  |Z| > 2.5   → ❌ DANGER — ห้ามเปิด (ใกล้ SL เกินไป)

  ★ ทำไมต้องมี Safe Zone:
  คนทำจริงเคยเปิดที่ Z = 3.607 ทั้งที่ SL = 3.5
  โดน SL ปิดใน 6 วินาที เกิดซ้ำ 48 ครั้ง เสียเงินจริง
```

## บทที่ 4: วิธีเข้าออเดอร์ (Entry)

### 4.1 เงื่อนไข Entry — ต้องผ่านทุกข้อ

```
☐ 1.  Correlation ≥ 0.80 (Rolling 90 วัน)
☐ 2.  Hurst Exponent < 0.50 (Mean-Reverting)
☐ 3.  Half-Life อยู่ในช่วง 5–30 วัน
☐ 4.  |Z-Score| ≥ 2.0
☐ 5.  Z-Score อยู่ใน Safe Zone (ไม่เกิน SafeMax)
☐ 6.  เหรียญทั้ง 2 มี Futures Contract จริง
☐ 7.  ราคาทั้ง 2 ตัว > 0
☐ 8.  β > 0, ≠ Infinity, ≠ NaN
☐ 9.  Position Size > 0, ≠ Infinity
☐ 10. ไม่มีคู่เดิมเปิดอยู่แล้ว (Dedup)
☐ 11. ไม่อยู่ใน Cooldown (เพิ่งปิดไป < 1 ชม.)
☐ 12. เหรียญเดียวกันไม่เกิน 2 คู่ (Concentration)
☐ 13. Volume เพียงพอ (> $20M ทั้ง 2 ตัว)

ผ่านทุกข้อ → เปิดได้
ไม่ผ่านแม้แต่ข้อเดียว → ห้ามเปิด
```

### 4.2 ทิศทางการเปิด

```
Z-Score > +2.0 (A แพงเกินไปเทียบ B):
  → Short A + Long B = "Short Spread"

Z-Score < -2.0 (A ถูกเกินไปเทียบ B):
  → Long A + Short B = "Long Spread"
```

### 4.3 Position Sizing

```
ขนาดพื้นฐาน: $500 ต่อขา (ปรับได้ใน Config)

Leg A: $500
Leg B: $500 × β

ตัวอย่าง: β = 1.3
  Leg A: Short ETH $500
  Leg B: Long SOL $650
  รวม: $1,150

ถ้าอยู่ใน Caution Zone: ลดเหลือ 50%
  Leg A: Short ETH $250
  Leg B: Long SOL $325

จำกัดต่อคู่: ไม่เกิน 5–10% ของพอร์ตทั้งหมด
```

### 4.4 Atomic Execution — เปิด 2 ขาต้องสำเร็จทั้งคู่

```
ขั้นตอน:
1. ส่ง Leg A → รอยืนยัน
2. Leg A สำเร็จ → ส่ง Leg B → รอยืนยัน
3. Leg B สำเร็จ → ✅ บันทึก DB ทันที

ถ้า Leg B ล้มเหลว:
  → Rollback: ปิด Leg A ทันที
  → ตรวจว่าปิดจริง (Post-Rollback Verification)
  → ถ้าปิดไม่ได้ → แจ้งเตือนด่วน ต้องจัดการด้วยมือ

★ ห้ามปล่อยให้มีขาเดียวค้างบน Exchange เด็ดขาด
  คนทำจริงเคยเจอ: Leg A สำเร็จ Leg B fail
  ไม่มี Rollback → ขาเดียวค้าง → เสียเงิน
```

### 4.5 Grace Period — หลังเปิด 5 นาที ห้าม SL

```
ทำไมต้องมี:
  เมื่อเปิดออเดอร์ Z-Score อาจกระโดดขึ้นเล็กน้อย
  ถ้าไม่มี Grace Period อาจโดน SL ทันทีใน 6 วินาที

กฎ:
  หลังเปิดออเดอร์ → ตั้ง Grace Period 5 นาที
  ภายใน 5 นาที → ห้าม Trigger Stop Loss
  แต่ Take Profit ยังทำงานปกติ
  และ Max Loss Stop (% ของพอร์ต) ยังเป็น Hard Limit
```

## บทที่ 5: วิธีออกออเดอร์ (Exit)

### 5.1 Take Profit

```
เงื่อนไข: Z-Score กลับมาใกล้ 0

ระดับ TP:
  Z กลับมาที่ 0      → ปิดทั้งหมด (กำไรเต็ม)
  Z กลับมาที่ ±0.5   → ปิดทั้งหมด (แนะนำสำหรับมือใหม่)
  Z กลับมาที่ ±1.0   → ปิดทั้งหมด (Conservative)

วิธี Partial TP:
  Z กลับมาที่ ±1.0  → ปิด 50%
  Z กลับมาที่ ±0.5  → ปิดอีก 25%
  Z กลับมาที่ 0     → ปิดที่เหลือ
```

### 5.2 Stop Loss — มี 4 ประเภท

```
SL ที่ 1: Z-Score Stop
  เงื่อนไข: |Z-Score| เกิน 3.0
  ความหมาย: Spread ถ่างออกหนักเกินคาด ความสัมพันธ์อาจเปลี่ยน
  ★ ต้องรอ Grace Period หมดก่อน (5 นาที)

SL ที่ 2: Time Stop
  เงื่อนไข: ถือนานเกิน 2 × Half-Life
  ตัวอย่าง: HL = 10 วัน → ตัดที่ 20 วัน
  ความหมาย: ถ้ากลับไม่ได้ภายใน 2× HL แปลว่าอาจไม่กลับแล้ว

SL ที่ 3: Correlation Break
  เงื่อนไข: Rolling Correlation ลดต่ำกว่า 0.50
  ความหมาย: ความสัมพันธ์พังแล้ว หลักการทั้งหมดไม่ใช้แล้ว

SL ที่ 4: Max Loss Stop
  เงื่อนไข: ขาดทุนเกิน 5% ของเงินที่จัดสรรให้คู่นี้
  ความหมาย: Hard Limit ป้องกันความเสียหายต่อพอร์ต
  ★ ข้อนี้ทำงานแม้อยู่ใน Grace Period
```

### 5.3 Emergency Exit

```
ออกทันทีเมื่อ:
  - เหรียญตัวใดตัวหนึ่งถูก Delist
  - มีข่าวร้ายแรง (Hack, Rug Pull)
  - Volume หายไปฉับพลัน (Liquidity Crisis)
  - Funding Rate สูงเกิน 0.1% ต่อ 8 ชม. (กินกำไร)
  - Exchange มีปัญหา (API ล่ม, ระงับถอน)
  - พบ Orphan Order (ขาเดียวค้าง)
```

### 5.4 Post-Close Verification

```
หลังปิดออเดอร์ทุกครั้ง:
  1. รอ 2 วินาที
  2. Query Exchange: มี Position เหลืออยู่ไหม
  3. ถ้า Position = 0 → ✅ ปิดสำเร็จ
  4. ถ้ายังมี Position → ส่งคำสั่งปิดอีกครั้ง
  5. ถ้ายังปิดไม่ได้ → แจ้งเตือน จัดการด้วยมือ

★ ต้องทำทุกครั้ง — คนทำจริงเคยเจอ:
  ส่งคำสั่งปิด แต่ไม่ตรวจว่าปิดจริง
  ผลคือ Position ค้างที่มองไม่เห็น
```

## บทที่ 6: ระยะเวลาถือ (Hold Duration)

### 6.1 ประเมินจาก Half-Life

```
Optimistic (เร็วสุด):  0.5 × Half-Life
Average (ค่ากลาง):      1.0 × Half-Life
Max (Time Stop):         2.0 × Half-Life

ตัวอย่าง: Half-Life = 10 วัน
  เร็วสุด: 5 วัน
  ค่ากลาง: 10 วัน
  นานสุด: 20 วัน (ถ้าเกินนี้ → ตัดทิ้ง)
```

### 6.2 ตามประเภทเหรียญ

| ประเภท | ตัวอย่าง | ถือประมาณ | เหตุผล |
|---|---|---|---|
| Large Cap × Large Cap | BTC/ETH, ETH/BNB | 7–14 วัน | เสถียร กลับช้าแต่แน่นอนกว่า |
| Large Cap × Mid Cap | ETH/SOL, BNB/AVAX | 5–10 วัน | ปานกลาง |
| Mid Cap × Mid Cap | LINK/DOT, AVAX/NEAR | 3–7 วัน | ผันผวนมากขึ้น |
| Small/Meme × Small/Meme | FLOKI/MEME, NOT/TURBO | 1–5 วัน | ผันผวนสูงมาก ทั้งกลับเร็วและพังเร็ว |

### 6.3 ข้อสังเกตจากระบบจริง

```
จากภาพระบบจริง: มี 4 คู่จาก ~7 คู่ที่โดน SL (อัตรา SL ≈ 57%)
ทั้ง 4 คู่เป็น Small/Meme Cap ทั้งหมด
→ Small Cap มีโอกาสพังสูง ควรเน้น Large-Mid Cap เป็นหลัก
→ Small Cap เทรดได้ แต่ลดขนาดลง 50%
```

## บทที่ 7: การคำนวณกำไรขาดทุน

### 7.1 สูตรกำไร

```
กำไร Leg A (Short):
  PnL_A = (ราคาเปิด - ราคาปิด) / ราคาเปิด × ขนาด USD

กำไร Leg B (Long):
  PnL_B = (ราคาปิด - ราคาเปิด) / ราคาเปิด × ขนาด USD

กำไรรวม = PnL_A + PnL_B
กำไรสุทธิ = กำไรรวม - ค่า Fee - ค่า Funding Rate
```

### 7.2 ค่าใช้จ่ายที่ต้องคำนึง

```
1. Trading Fee: ~0.04-0.06% ต่อ Order (OKX Futures)
   เปิด 2 ขา + ปิด 2 ขา = 4 orders
   ต้นทุน Fee ≈ 0.04% × 4 × ขนาดรวม = 0.16% ของขนาดรวม

2. Funding Rate: จ่ายทุก 8 ชม.
   ขึ้นอยู่กับสภาพตลาด
   ตลาดขาขึ้น: Funding Rate สูง → ถือ Short แพง
   ตลาดขาลง: Funding Rate ต่ำหรือติดลบ → ถือ Short ถูก

3. Slippage: ราคาจริงอาจต่างจากที่คำนวณ
   เหรียญ Volume สูง: < 0.05%
   เหรียญ Volume ต่ำ: 0.1-0.5%
```

### 7.3 ตัวอย่างคำนวณจริง

```
เปิด: Short ETH $500 ที่ $3,600 / Long SOL $650 ที่ $140
ปิด: ETH $3,450 / SOL $148

Leg A (Short ETH):
  PnL = (3,600 - 3,450) / 3,600 × $500 = +$20.83

Leg B (Long SOL):
  PnL = (148 - 140) / 140 × $650 = +$37.14

กำไรรวม: $20.83 + $37.14 = $57.97

หัก Fee: 0.04% × 4 × $1,150 = $1.84
หัก Funding (สมมติถือ 7 วัน): ~$5.00
หัก Slippage: ~$1.50

กำไรสุทธิ: $57.97 - $1.84 - $5.00 - $1.50 = $49.63
ROI: $49.63 / $1,150 = 4.3% ใน 7 วัน
```

## บทที่ 8: การบริหารความเสี่ยง

### 8.1 ความเสี่ยงทั้งหมดที่ต้องรู้

| ความเสี่ยง | ระดับ | คำอธิบาย | วิธีป้องกัน |
|---|---|---|---|
| Spread ไม่กลับมา | สูง | ความสัมพันธ์พังถาวร | Z-Score SL + Time Stop + Corr Break |
| Orphan Order | สูงมาก | ขาเดียวค้าง Exchange | Atomic Execution + Rollback |
| ออเดอร์ซ้ำ | สูง | เปิดคู่เดิมซ้ำหลายครั้ง | Dedup 5 ชั้น |
| SL ทันที | สูง | เปิดใกล้ SL เกินไป | Safe Entry Zone + Grace Period |
| ข้อมูลหาย | สูงมาก | Restart แล้วทุกอย่างหาย | DB-First + Rebuild Cache |
| Config ผิด | ปานกลาง-สูง | ใช้ค่าที่ไม่ได้ Backtest | Backtest ก่อนใช้ |
| Concentration | สูง | เหรียญเดียวกันเปิดหลายคู่ | จำกัด 2 คู่ต่อเหรียญ |
| Funding Rate สูง | ปานกลาง-สูง | กินกำไรหมด | เช็คก่อนเข้า ออกเมื่อ > 0.1%/8h |
| Liquidity ต่ำ | สูง | เข้า-ออกยาก Slippage สูง | Volume > $20M + Size < 1% ของ Vol |
| Leverage Liquidation | สูงมาก | ถูก Liquidate ก่อน SL | จำกัด Leverage ≤ 3x |
| β เปลี่ยน | ปานกลาง | Position ไม่ Balance | ตรวจทุก 2 สัปดาห์ |
| Exchange ล่ม | ปานกลาง | ปิดออเดอร์ไม่ได้ | เผื่อ Buffer SL + Exchange สำรอง |

### 8.2 กฎบริหารเงิน (Money Management)

```
1. ขนาดต่อคู่:    $500 ต่อขา (ไม่เกิน 10% ของพอร์ต)
2. จำนวนคู่สูงสุด: 3–5 คู่พร้อมกัน
3. Max Drawdown:   หยุดเทรดถ้าขาดทุนรวมเกิน 15% ของพอร์ตเริ่มต้น
4. กระจายความเสี่ยง: ห้ามใช้เหรียญเดียวกันเกิน 2 คู่
5. Leverage:       ไม่เกิน 2–3x
6. Paper Trade:    ทดสอบอย่างน้อย 2 สัปดาห์ก่อนใช้เงินจริง
7. เริ่มเล็ก:      เริ่มด้วย $100 ต่อขา แล้วค่อยเพิ่ม
```

### 8.3 ระดับความเสี่ยงรวมของกลยุทธ์

**ปานกลาง-สูง**

Pairs Trading ในตลาด Crypto มีความเสี่ยงสูงกว่าในตลาดหุ้น เพราะ Crypto ผันผวนสูงกว่า ความสัมพันธ์ระหว่างเหรียญเปลี่ยนเร็ว และ Liquidity ของ Altcoin หลายตัวยังไม่เพียงพอ นอกจากนี้ ความเสี่ยงจากระบบ (Operational Risk) อาจร้ายแรงกว่าความเสี่ยงจากตลาดเสียอีก จากประสบการณ์จริง 31% ของ Bug ทั้งหมด (8 จาก 26 ตัว) เป็นระดับ Critical ที่กระทบเงินจริง

## บทที่ 9: Dedup 5 ชั้น — ป้องกันเปิดซ้ำ

### ทำไมต้องถึง 5 ชั้น

```
คนทำจริงเจอ: Scanner ส่งสัญญาณคู่เดิมซ้ำทุกรอบ
Auto-Trade เปิดคู่เดิมซ้ำ 3-4 ครั้ง
Telegram ส่ง Alert ซ้ำ

แก้: ตรวจ 5 ชั้น ก่อนเปิดทุกครั้ง

ชั้น 1 — Redis:    คู่นี้มีใน Cache ไหม (เร็วสุด)
ชั้น 2 — Database: คู่นี้มี Record Open ใน DB ไหม
ชั้น 3 — Exchange: มี Position จริงบน OKX ไหม
ชั้น 4 — Cooldown: เพิ่งปิดคู่นี้ไปภายใน 1 ชม. ไหม
ชั้น 5 — Concentration: เหรียญเดียวกันเกิน 2 คู่ไหม

ไม่ผ่านแม้แต่ชั้นเดียว → ห้ามเปิด
```

## บทที่ 10: Reconciliation — เทียบ DB กับ Exchange

```
ทุก 5 นาที ระบบจะเช็ค:

1. ดึง Position ที่เปิดอยู่จาก DB
2. ดึง Position ที่เปิดอยู่จาก Exchange
3. เทียบกัน:

  ✅ ตรงกัน (อยู่ใน DB + อยู่บน Exchange) → OK

  ⚠️ Ghost (อยู่ใน DB แต่ไม่อยู่บน Exchange)
     → อัปเดต DB ว่าปิดแล้ว + แจ้งเตือน

  🔴 Orphan (อยู่บน Exchange แต่ไม่อยู่ใน DB)
     → แจ้งเตือนทันที
     → ห้าม Auto-Close เด็ดขาด!
     → ต้องตรวจสอบและจัดการด้วยมือ

★ ทำไมห้าม Auto-Close Orphan:
  คนทำจริงเคยเจอ: Server restart → Memory หาย
  → Orphan Detection เห็นว่ามี Position แต่ไม่มีใน Memory
  → Auto-Close ปิดออเดอร์ที่ถูกต้องทิ้ง → สูญเสียเงินจริง
```
